CASE STUDY

WHAT DO YOU MEME?TM MOBILE GAME

ABOUT THIS PROJECT

In this project, we were tasked with converting What Do You Meme?™ (WDYM), a popular tabletop game for adults, into a mobile game and creating a new product that would engage new and old fans.

The interactive prototype we presented to our stakeholders gave them the ability to try out these features: 1) signup, 2) update their profile, 3) change the music setting, and 4) learn the game through a quick tutorial.

VIEW PROTOTYPE

PROBLEM & SOLUTION

The stakeholders presented us with this question: How would we transform WDYM tabletop game into a mobile game while preserving and incorporating its original game mechanics? It was important to the stakeholders that their users experienced familiarity and a seamless transition in going from one format to the other.

Based on our design research and user tested results, we created a data-driven blueprint for building WDYM mobile game. In general, we were able to stay true to the original tabletop game and also introduced a few new features, such as a timer and a status update window, which allowed us to “translate” certain gameplay experiences into a better fit for the mobile game format.

COMPETITIVE ANALYSIS

Based on the research data we collected, we mapped out some of the differences in user experience between tabletop and mobile games. We illustrated some of our key findings in the chart below.

TABLETOP GAMES

  • Face-to-face human interaction
  • Tactile feel of game board and pieces
  • Physical actions of moving game pieces, writing notes and points, shuffling cards
  • Long instruction manuals for complex games
  • Longer game sessions (over many hours)
  • Can be bought in stores or online

MOBILE GAMES

  • Play with anyone, anywhere in the world
  • No wear ‘n tear or missing game pieces
  • Automation of tracking, player order, actions, points, cards, and game pieces
  • Computer guided practice game or tutorial
  • Shorter game sessions (until battery dies)
  • Can download and play immediately

DESIGN

In order to create a seamless experience going from the tabletop game to the mobile game, it was important that we kept the branding and visual style of WDYM as consistent as possible. We curated a moodboard of popular memes and images from the WDYM website.




For type, we chose Helvetica, which is used in the WDYM Caption Cards, and Impact, which is considered the defacto meme font. For the color palette, we sampled the colors from the WDYM logo, which allowed us to create a cohesive style between the tabletop and mobile versions of WDYM. We also created a pattern library of components that we used to build the mobile app, which allowed us to maintain consistency among all elements.


INFORMATION ARCHITECTURE

Using a process that was loosely inspired by the 6-8-5 sketching game, I was able to brainstorm some initial ideas, which helped me to map out and create user stories, user flows, and the script for the mobile game tutorial. With each iteration of sketches, the user experience and layout of features began to take more shape and form, as old questions were answered and new questions came up. For example, where should the card decks go? Where should the avatars be placed? How large or small should we make the profile image? How would player’s differentiate between the judge’s deck or the player’s deck of cards? How would players keep track of the score? How would we prompt the player when it’s his turn to be the judge?


USABILITY TESTING

We engaged in rapid prototyping and iterated constantly as we built the prototype from wireframes to mockups, testing each design frequently and breaking things as quickly and as often as we could. In total, we cycled through roughly four iterations of design and testing before arriving at our current prototype.

Using Google Hangouts and screen sharing, I followed a usability test script and remotely observed three users in real-time as they completed their assigned tasks in using the prototype. Afterward, I compiled their comments and feedback into a Google Doc.

RAPID PROTOTYPING

Here are some examples of how the mobile game features evolved through the design, testing, and iteration process:

BACKGROUND: We combined some ideas from our moodboard in various ways and sent them through a preference test. Based on users’ responses, we created our final version of the mobile game’s background.

BUTTONS: We originally designed a “drag + drop” method for users to draw new cards from the Photo Card or Caption Card deck. In the end, we decided to use buttons they could tap.

CAPTIONS: We designed the Caption Cards to overlay their caption on the Photo Card for a few reasons. Given that the mobile game format lives in the digital world, we wanted to replicate the experience of creating a meme, exactly as you would in real life. Also, given the wide age demographic of our users and our emphasis on accessibility, we felt that seniors would most likely have more difficulty reading smaller print (of the Caption Cards in the card tray) and would appreciate the caption displayed in a larger font over a Photo Card.

GAME STATUS: In any tabletop game, there’s always a person that keeps track of the game status, whose turn it is, and what actions need to happen next. We solved that problem by introducing a status window that easily allows us to update all the players. Initially, we added an input field of emojis to allow players to like or respond to a caption that they liked. However, during user testing, we discovered that all 100% of our users disliked this feature, citing that it was confusing or that they probably wouldn’t use it. Consequently, we removed the feature.

TUTORIAL: We tested the tutorial script and colored bubbles until we achieved a good balance of brevity and depth that would not bore current fans or confuse new fans. [ Preference Test #1 | Preference Test #2 ]

TIMER: In Exploding Kittens mobile game, there was a gauge that moved based on a player’s likelihood of drawing the exploding kitten card (the one card players avoided at all cost). This feature inspired us to introduce a timer to WDYM mobile game. With a timer, the mobile game created a greater sense of urgency allowing players to experience the adreneline-inducing pace of the tabletop game. Without the timer, we noticed that the players and judge took much longer to read and submit their Caption Cards. Once we added a timer, our users commented that they felt less distracted and more engaged in playing WDYM mobile game.
